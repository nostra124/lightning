# Lightning cluster integration — `lightning serve` HTTP endpoint

## Description

**As a** cluster operator running the `nostra124/cluster`
stack with apache
**I want** `lightning serve` to expose a minimal HTTP endpoint
(`POST /invoice`, `GET /pay/<bolt11>`, `GET /address/<user>`)
that apache proxies
**So that** the cluster's payment story is exactly the
educational `lightning` surface, no custodial wrapper.

This is the lightning-side counterpart of cluster FEAT-079
(apache + lightning).

## Implementation

1. **`lightning serve [--listen 127.0.0.1:NNNN]`** — runs a
   small bash-only HTTP server (busybox `httpd` or `socat`
   loop) that maps incoming requests to verbs:
   - `POST /invoice` body `{"msat": N, "memo": "..."}` →
     `lightning invoice` → BOLT-11 string
   - `GET /pay/<bolt11>` → `lightning pay <bolt11>` (gated by
     a shared secret; default off)
   - `GET /address/<user>` → Lightning Address `.well-known`
     response for hosted users (FEAT-176)
2. **systemd unit** generated by `lightning daemon install`
   (FEAT-183) gains a `--serve` flag.
3. **apache snippet** under `share/doc/lightning/cluster/` —
   ProxyPass config the cluster operator drops into vhost.
4. **Auth**: bearer token from `secret get lightning.serve.token`;
   no-token mode refuses any verb that moves funds.

## Acceptance Criteria

1. `lightning serve` plus a curl `POST /invoice` returns a
   parseable BOLT-11.
2. `GET /address/alice` returns LUD-16 JSON when FEAT-176 is
   present and alice is configured.
3. Apache snippet renders a cluster-side
   `/lightning/invoice` endpoint.
4. Auth bypass returns 401 for pay endpoints.

## Milestone

Post-1.0. Lands together with cluster FEAT-079 when both
sides are ready.
