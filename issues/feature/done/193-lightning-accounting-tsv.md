# Lightning accounting — small SQLite store

## Description

**As a** Lightning user
**I want** a small SQLite store of accounts, ledger entries,
and outstanding invoices, plus a few view verbs
**So that** alice's shell and the Apache CGI scripts
(FEAT-196) can read and write concurrently without
file-locking dances, balance queries are joins instead of
awk passes, and "running a small bank" gets real
referential integrity.

Replaces the rec-format `history` from the FEAT-174 draft
and the TSV ledger from the first cut of this ticket. Still
small — five tables, no triggers, no views, no migrations
in v1. The shell verbs default to TSV stdout so `awk` /
`cut` / spreadsheet workflows survive; SQLite is the
storage, not the interface.

Soft dep on `sqlite3` (system CLI) declared at
`.rpk/depends/sqlite3`. Available out of the box on every
target distro.

## Implementation

### File

`~/.lightning/wallet/<name>/state.db` — SQLite 3.x, WAL
mode (allows concurrent reads + one writer, which is the
common case: alice's shell reads, the CGI write).

A companion `state.sql` lives next to it; it's the
`sqlite3 state.db .dump` output, regenerated by a
pre-commit git hook (see FEAT-174). Git tracks the SQL
dump — readable diffs, restorable on `wallet pull`. The
binary `state.db` is git-ignored.

### Schema

Five tables, all in one file:

    CREATE TABLE accounts (
        name        TEXT    PRIMARY KEY,
        description TEXT    NOT NULL DEFAULT '',
        limit_sat   INTEGER,                    -- NULL = no limit
        overdraft   TEXT    NOT NULL DEFAULT 'deny'
                                                -- deny|warn|allow
                                                -- (FEAT-195)
    );

    CREATE TABLE ledger (
        id           INTEGER PRIMARY KEY,
        ts           TEXT    NOT NULL,          -- RFC-3339 UTC
        account      TEXT    NOT NULL DEFAULT '-'
                             REFERENCES accounts(name),
        direction    TEXT    NOT NULL,          -- in|out|fee|forward
        amount_msat  INTEGER NOT NULL,          -- signed
        peer         TEXT    NOT NULL DEFAULT '-',
        payment_hash TEXT    NOT NULL DEFAULT '-',
        message      TEXT    NOT NULL DEFAULT '',
                                                -- visible to both peers
        note         TEXT    NOT NULL DEFAULT ''
                                                -- local-only
    );

    CREATE TABLE invoices (
        bolt11       TEXT    PRIMARY KEY,
        payment_hash TEXT    NOT NULL,
        account      TEXT    NOT NULL DEFAULT '-'
                             REFERENCES accounts(name),
        amount_msat  INTEGER NOT NULL,
        expiry       TEXT    NOT NULL,
        message      TEXT    NOT NULL DEFAULT '',
        state        TEXT    NOT NULL DEFAULT 'pending'
                                                -- pending|settled|expired
    );

    CREATE TABLE channel_notes (
        channel_id TEXT PRIMARY KEY,
        note       TEXT NOT NULL
    );

    CREATE TABLE users (                        -- Lightning Address
        user        TEXT    PRIMARY KEY,        -- e.g. 'alice'
        account     TEXT    NOT NULL
                            REFERENCES accounts(name),
        min_sat     INTEGER NOT NULL DEFAULT 1,
        max_sat     INTEGER NOT NULL DEFAULT 100000000,
        comment_max INTEGER NOT NULL DEFAULT 256
    );

Notes:

- `ledger.payment_hash` is `-` for non-payment rows (fees,
  forwards aggregated). For payment rows it's the hex hash;
  not unique because a settled invoice and its fee row share
  the hash.
- `users` replaces FEAT-176's `users.tsv`. Same per-wallet
  scope as the rest — one DB per wallet.
- `accounts.overdraft` and `accounts.limit_sat` are the
  FEAT-195 fields.

### Verbs

    lightning ledger list   [--account <n>] [--since <date>]
                            [--limit N]
    lightning ledger sum    [--by account|day|month|year]
                            [--account <n>]
    lightning ledger balance [<account>]
    lightning ledger annotate <payment_hash> <note>
    lightning ledger export csv|jsonl|tsv > out

All are thin shells over `sqlite3 state.db <query>`. Default
output is TSV (tab-separated, header row).

`ledger annotate` is a single `UPDATE ledger SET note = ?
WHERE payment_hash = ?`. No append-and-supersede dance.

### Writers

Two writers in production:

- alice's shell (via `lightning ...` verbs)
- the CGI scripts (which sudo to alice and re-enter the same
  verbs — see FEAT-196's privileged-hop section)

Both go through the `lightning` CLI. There is no direct
SQLite access from www-data. The CLI takes a brief
`BEGIN IMMEDIATE` transaction per ledger write; collisions
back off and retry with jitter (SQLite's busy_timeout set to
5000ms).

### What this does NOT do

- No migrations in v1. Schema changes ride along with the
  major version bump that introduces them; the install
  script refuses to run against an older schema.
- No triggers, no views, no foreign-key cascades. Verbs
  enforce invariants in Python / bash; SQL stays declarative
  and readable.
- No multi-statement transactions exposed to scripts. Each
  verb's write is a single `INSERT` or `UPDATE`.

## Acceptance Criteria

1. `lightning wallet init` creates `state.db` with the five
   tables above.
2. Every `lightning pay` and `lightning invoice settled`
   appends exactly one `ledger` row.
3. `lightning ledger balance alice` returns the same number
   as `SELECT SUM(amount_msat) FROM ledger WHERE account =
   'alice'`.
4. `lightning ledger sum --by month` returns per-month
   totals in TSV.
5. `lightning ledger export csv` produces a spreadsheet-
   ready CSV.
6. `sqlite3 state.db 'SELECT * FROM ledger'` works without
   any `lightning` tool installed — the file is plain
   SQLite.
7. `lightning ledger annotate <hash> <note>` updates the
   row; subsequent `ledger list` reflects it.
8. Concurrent writes from alice's shell and a CGI
   `sudo`-bridged invocation succeed (SQLite WAL +
   `BEGIN IMMEDIATE`).
9. Pre-commit hook regenerates `state.sql`; the binary
   `state.db` is gitignored.
